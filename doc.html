Documentación de Ingeniería Avanzada

La House Pricing Extractor API es un motor de procesamiento asíncrono diseñado para la orquestación de flujos de Web Scraping complejos, evasión de Web Application Firewalls (WAF) y consolidación de datos ETL (Extract, Transform, Load).

1. Arquitectura del Sistema

El sistema implementa un patrón Arquitectura Hexagonal (Puertos y Adaptadores) adaptado para tareas asíncronas de larga duración.

+----------------------+ +-----------------------------------------+ +------------------------+ | Capa Web | | Capa de Aplicación | | Capa de Infra / BD | | (FastAPI) | | (Pipeline) | | | | | | +---------+ +--------+ +--------+ | | +----------------+ | | [POST /upload] ----> | ---> | | Paso 0 |-->| Paso 1 |-->| Paso 2 | | ----> | | MySQL Server | | | | | | (Auth) | | (Parse)| | (Scrap)| | | | (Transaccional)| | | [GET /status] <---- | <--- | +---------+ +--------+ +--------+ | | +----------------+ | | | | | | | | | | +----------------------+ | v v v | +------------------------+ | [ Eventos de Progreso Compartido ] | +-----------------------------------------+

Decisiones de Diseño Clave:

  • Desacoplamiento de la petición HTTP: Las peticiones de subida (/upload-process) responden inmediatamente con un task_id (UUID). La lógica pesada se delega a BackgroundTasks de FastAPI.
  • Inmutabilidad del Pipeline: Cada "Paso" recibe datos estructurales (JSON/Dict) y retorna un estado modificado, operando como funciones puras siempre que es posible para facilitar el testing y aislar fallos.

2. Patrones de Concurrencia y Paralelismo

El sistema procesa lotes (batches) de propiedades utilizando paralelismo para reducir los tiempos de I/O (descargas y navegación web).

Implementación: ThreadPoolExecutor

Tanto en el Paso 0 (Descargas vía requests) como en el Paso 2 (Navegación vía Selenium), se utiliza concurrent.futures.ThreadPoolExecutor. Se descartó asyncio en el core del scraping debido a la naturaleza síncrona y bloqueante de undetected_chromedriver y las librerías de parseo de PDF.

# Ejemplo conceptual de partición de carga (Chunking)
WORKERS = 5
chunk_size = math.ceil(total_propiedades / WORKERS)
chunks = [lista_propiedades[i:i + chunk_size] for i in range(0, total, chunk_size)]

with ThreadPoolExecutor(max_workers=WORKERS) as executor:
    futures = [executor.submit(worker_function, id, chunk, cancel_event) for chunk in chunks]
    for future in as_completed(futures):
        resultados.extend(future.result())
Precaución de Recursos (Thread Safety): En el Paso 2, al levantar 5 instancias de Chromium simultáneas, el consumo de RAM puede escalar a ~2.5GB. El servidor debe ser dimensionado en consecuencia. Las devoluciones de llamadas (callbacks) de progreso utilizan threading.Lock() para prevenir condiciones de carrera al actualizar variables globales de estado.

3. Estrategia de Evasión (WAF Bypass)

El sistema se enfrenta a **Cloudflare Turnstile**, una protección diseñada para bloquear tráfico automatizado evaluando la huella digital del navegador (TLS/JA3) y la biometría de interacción.

El Patrón "Abrepuertas" (Paso 0)

  1. Ejecución con Monitor Virtual (Xvfb): Cloudflare penaliza severamente la bandera --headless. Para correr en un servidor Linux, se utiliza pyvirtualdisplay. Esto engaña al WAF haciéndole creer que existe hardware de renderizado conectado (GPU/Display).
  2. Parcheo de Driver: Se utiliza undetected_chromedriver, el cual recompila el binario subyacente de chromedriver para eliminar variables clave (como navigator.webdriver = true).
  3. Inyección Biométrica: Para superar el análisis heurístico, el login no se inyecta por JavaScript ni con send_keys() instantáneo. Se utiliza ActionChains para trazar rutas de mouse y un bucle de retraso aleatorio inter-tecla (50-150ms) simulando latencia humana.
  4. Transferencia de Sesión (Handoff): Una vez Cloudflare emite las cookies de validación (sessionid y csrftoken), la instancia pesada de Selenium se destruye. Las cookies se inyectan en sesiones ultraligeras de requests.Session() que realizan el trabajo de descarga a máxima velocidad sin ser re-evaluadas por el WAF.

4. Pipeline de Ejecución Deep-Dive

Paso 1: Extracción Semántica Dual (HTML vs PDF)

Implementa un patrón de degradación elegante (Graceful Fallback).

  • El Motor Principal (DOM Parsing): paso1_html_hp.py. Utiliza BeautifulSoup. Es determinista (O(n) de lectura de árbol). Localiza nodos por IDs específicos (#section-resumen). Costo computacional: Marginal.
  • El Motor Fallback (Análisis Espacial OCR): paso1_pdf_hp.py. Si el HTML cambia su estructura o no está disponible, se utiliza pdfplumber. Este motor no lee texto plano, sino que calcula distancias euclidianas (Eje Y/X) entre los bounding boxes de las palabras ("N°", "Material") para reconstruir tablas complejas que se rompen en multilínea. Identifica hipervínculos midiendo la proximidad de la capa de anotación URI a los números de rol.

Paso 3: Generación Estructural de Reportes

Transforma un JSON anidado en un modelo relacional plano mediante pandas. Utiliza openpyxl para manipular el DOM del archivo .xlsx resultante, inyectando estilos de cabecera y formateando cadenas de texto que comiencen con http como hipervínculos clickeables.

5. Modelo de Datos e Inyección ETL (Paso 4)

El módulo paso4_hp.py actúa como un servicio ETL para poblar el Data Warehouse de la aplicación.

Estrategia Transaccional

Para asegurar el principio ACID (Atomicidad, Consistencia, Aislamiento, Durabilidad), toda la inyección de una propiedad se realiza bajo una única transacción de base de datos.

try:
    conn.start_transaction()
    # 1. Insert/Update Tabla Principal
    # 2. Insert Tablas Relacionales (Deudas, Roles, Construcciones)
    conn.commit()
except mysql.connector.Error as err:
    conn.rollback() # Si falla la línea 2, se deshace la línea 1
    manejar_error(err)

Manejo de Conflictos (Upserts)

Para evitar colisiones de llaves primarias o índices únicos (ej. El número de Rol), se utiliza el paradigma ON DUPLICATE KEY UPDATE de MySQL, lo que permite que el scraper actualice registros antiguos si se vuelve a procesar una propiedad existente, sin generar errores 1062.

6. Gestión de Estado y Señales IPC (API)

La API (FastAPI) debe rastrear procesos que corren en hilos separados. Esto se logra mediante el diccionario global TASKS.

Atributo Tipo Propósito
status String Máquina de estados: processing, completed, failed, cancelled.
progress Integer Porcentaje (0-100) actualizado mediante callbacks inyectados en las funciones de los workers.
cancel_event threading.Event() Primitiva de sincronización (IPC). Cuando el usuario envía una petición de cancelación, se invoca event.set(). Los loops internos en los Pasos 0, 1, 2 y 3 revisan if event.is_set(): break en cada iteración, permitiendo un cierre elegante sin matar el proceso a la fuerza.
file_path String Referencia al archivo generado en disco para su posterior descarga (FileResponse).

7. Infraestructura y Despliegue (Production Readiness)

Script de Inicialización (Systemd)

Para entornos de producción (Ubuntu/Debian), el servidor Uvicorn (FastAPI) nunca debe ejecutarse directamente en la terminal. Se debe crear un servicio systemd para garantizar el reinicio automático ante caídas.

# /etc/systemd/system/housepricing-api.service
[Unit]
Description=House Pricing FastAPI Engine
After=network.target

[Service]
User=tu_usuario
Group=www-data
WorkingDirectory=/var/www/house-pricing-api
Environment="PATH=/var/www/house-pricing-api/env/bin"
# Ejecución vía Uvicorn (Recomendado sobre python server.py)
ExecStart=/var/www/house-pricing-api/env/bin/uvicorn server:app --host 0.0.0.0 --port 3000 --workers 1
Restart=always

[Install]
WantedBy=multi-user.target

Monitor de Salud (Garbage Collection)

Las herramientas de Headless Browsing son notorias por crear fugas de memoria (Memory Leaks). Si un worker falla catastróficamente y no alcanza el bloque finally: driver.quit(), el proceso chrome queda "zombie". El script utils.py lee el árbol de procesos del sistema operativo (vía psutil) buscando procesos huérfanos asociados al PID padre para registrarlos en los logs y alertar sobre la necesidad de reciclaje del servidor.
